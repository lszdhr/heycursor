## 通用礼节 (General Etiquette)

- 优先保证代码简洁易懂。
- 别搞过度设计，简单实用就好。
- 写代码时，要注意复杂度，函数尽量小，尽量可以复用，尽量不写重复代码。
- 写代码时，注意模块设计，尽量使用设计模式。
- 给我解释代码的时候，最好不要太多专业术语。最好有图（mermaid风格）
- 帮我实现的时候，需要给出原理，并给出执行步骤，最好有图（mermaid风格）
- 改动或者解释前，最好看看所有代码，不能偷懒。
- 改动前，要做最小化修改，尽量不修改到其他模块的代码

# 实验性规则 (Experimental Rule)

当你被要求修复一个 Bug 时，请遵循以下步骤：

1. **理解问题 (Understand):** 仔细阅读 Bug 描述和相关代码，复述你对问题的理解。
2. **分析原因 (Analyze):** 提出至少两种可能的根本原因。
3. **制定计划 (Plan):** 描述你打算如何验证这些原因，并给出修复方案。
4. **请求确认 (Confirm):** 在动手修改前，向我确认你的计划。
5. **执行修复 (Execute):** 实施修复。
6. **审查 (Review):** 查看自己的修改有没有问题。
7. **解释说明 (Explain):** 解释你做了哪些修改以及为什么。
   Always respond in 中文

## 核心规则

1. **完整输出** - 输出完整代码和答案，禁止使用省略号或"代码过长不展示"
2. **持续执行** - 复杂任务尽量一次性完成，不要频繁中断或反复追问是否继续

## 反偷懒规则 (Anti-Laziness)

### 第一梯队：强制执行

1. **先读后改** - 修改任何文件前，必须先用 read_file 读取完整内容，禁止凭记忆或上下文摘要修改。未读取就编辑视为违规。
2. **禁止省略占位** - 代码中禁止出现 `// ... rest unchanged`、`// 其他逻辑同上`、`/* 省略 */` 等占位写法，必须写出完整代码。
3. **Checklist 必须附证据** - 自检 checklist 每一项打勾时，必须附带具体的文件名 + 行号或搜索结果作为证据，不能只写"已确认"。

### 第二梯队：约束推理过程

4. **禁止假设，必须验证** - 不确定某个函数、变量、文件是否存在时，必须用 grep_search 或 code_search 确认，禁止凭猜测下结论。
5. **改公共模块前列影响范围** - 修改 composable、utils、云函数等公共模块前，先用 grep 搜索所有调用方，列出影响范围，确认改动不会破坏其他模块。
6. **一次 edit 只做一件事** - 每次 edit 调用只解决一个逻辑变更，不要在一次 edit 中混合多个不相关的修改。

## Spec 自动同步规则

7. **功能变更必须同步 spec** - 任何涉及用户可感知功能的代码改动（新增功能、修改行为、新增数据字段/集合），完成后必须同步更新 `.spec/` 目录下对应的 requirements.md 和 design.md。纯样式调整、重构、性能优化不需要。
8. **补录格式** - 同步时在对应 AC 前标注 `[补录 YYYY-MM-DD]`，一行写清楚即可，不需要重走 spec-driven-dev 流程。
9. **完成时主动汇报** - 任务结束向用户汇报时，如果有 spec 同步，列出本次补录了哪些 AC；如果判断不需要同步，说明原因。

## 命令执行规则

10. **禁止终端内联长脚本** - 禁止在终端中直接执行多行内联代码（如 `python3 -c "..."` 包含多行逻辑）。超过 3 行的逻辑必须先写入脚本文件（如 `scripts/xxx.py`），再通过 `python3 scripts/xxx.py` 执行。执行完成后如果是一次性脚本，提示用户是否删除。
